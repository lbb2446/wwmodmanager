# 开发约束与规范
- 业务需求文档引用：所有开发变更必须参考 docs/需求文档/业务需求.md 的明确需求点与验收标准。
- 代码结构约束：仓库中 app.py 只能存在一个，且应位于仓库根目录；如发现不符，提交/合并将被拒绝。
- CI/PR 约束：通过 GitHub Actions 规则，未满足约束的提交将被阻止。
- 变更记录：每次功能修改需更新变更日志和需求映射，确保可追溯性。
- 分支命名：feat/、fix/、docs/、refactor/、chore/ 等前缀，后续与任务编号结合使用。
- 测试要求：新增功能应具备单元测试/集成测试；测试覆盖率目标按团队约定执行。
- 安全性：代码中避免注入、文件路径绕过等风险；对外部输入要做严格校验。
- 性能与资源：对图片下载、缓存、资源定位等可能的瓶颈点进行性能评估。
- 文档化：实现前撰写设计文档，完善 API/数据模型文档。
